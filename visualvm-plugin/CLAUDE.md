# visualvm-plugin

## Overview
- VisualVM console plugin (NetBeans Platform module, code-name-base `org.perfmon4j.visualvm`)
  exposing live perfmon4j interval/snapshot monitor browsing, ad-hoc real-time charting, and
  on-demand thread-trace scheduling, all over the agent's RMI remote-management interface
  (`base/src/main/java/org/perfmon4j/remotemanagement/`, consumed here via
  `org.perfmon4j.remotemanagement.intf.RemoteManagementWrapper`).
- Historically built for years as an Ant-driven NetBeans IDE module project
  (`netbeans/Perfmon4jMonitor/` + a `Perfmon4jSuite` wrapper), which could only be built from
  inside a NetBeans IDE with a locally-registered `VisualVM_1.3.8` Platform Manager alias
  (`nbplatform.active`) — never automatable outside the IDE, and never wired into CI. This
  directory is a from-scratch Maven port of that same source (same package, same classes): built
  with `nbm-maven-plugin` against the modern VisualVM plugin API, which — since VisualVM's
  GraalVM rebrand — is published to Maven Central. Nothing about the plugin's functionality
  changed; only the build.
- Not part of the root `pom.xml` reactor or the GitHub Actions workflow, like `hawtio-plugin/`
  and `stress-test/` — independent tooling, built and versioned standalone (own `1.0-SNAPSHOT`
  line, unrelated to the core `perfmon4j` release cycle).
- A separate initiative has since shipped a `RemoteManagement` JMX MBean (`base`) exposing this
  same RMI-based data over Jolokia/Hawtio instead, since RMI isn't reachable from a browser — see
  `hawtio-plugin/CLAUDE.md` and `hawtio-plugin/ROADMAP.md`. That doesn't change anything here —
  this module remains the RMI-based desktop UI and is fully functional against any running
  perfmon4j agent with its RMI listener enabled. `LEGACY_VISUALVM_FEATURES.md` documents this
  plugin's use cases in detail, written while scoping that Hawtio work.
- See `ROADMAP.md` for what's next (signing the `.nbm`, modern charting options, release/snapshot
  distribution) and why.

## Architecture & Patterns
- `Installer.java` — `ModuleInstall` entry point; registers/unregisters
  `Perfmon4jMonitorViewProvider` when the module loads/unloads.
- `Perfmon4jMonitorViewProvider.java` / `Perfmon4jMonitorView.java` — a
  `DataSourceViewProvider`/`DataSourceView` that adds a "Perfmon4j" tab to VisualVM's per-
  `Application` view whenever `Perfmon4jModel.getModelFor(app)` detects the attached JVM is
  perfmon4j-instrumented.
- `Perfmon4jModel.java` — auto-detects a perfmon4j RMI listener port on the attached
  `Application` (via JMX, the Attach API, or the Serviceability Agent, falling back to a regex
  over the raw JVM args), then opens an RMI connection through `RemoteManagementWrapper`.
- `MainWindow.java` (+ `.form`) — the main UI: monitor tree browsing, the chart panel, and the
  thread-trace tabs.
- `chart/` — `DynamicTimeSeriesChart`/`FieldManager`/`FieldElement` (JFreeChart-based scrolling
  time-series charting), `ChartElementsTable`/`TextElementsTable` (per-field controls),
  `ThreadTraceList`/`ThreadTraceTable` (async thread-trace queue), `SelectColorDlg`/
  `SelectFieldDlg` (add-field dialogs).
- `.form` files are NetBeans GUI Builder metadata only, editable if reopened in the NetBeans
  IDE — not a Maven build input. The paired `.java` file's already-generated `initComponents()`
  method is what actually compiles and runs.
- `src/main/nbm/manifest.mf` supplies the hand-written `OpenIDE-Module-Install`/`-Layer`/
  `-Localizing-Bundle`/`-Requires` entries that `nbm-maven-plugin` can't auto-derive (see "The
  manifest wiring" below); everything else in the final manifest (code-name-base, spec version,
  module dependencies, bundled-library classpath) is generated by the plugin from `pom.xml`.

## Stack Best Practices
- **`netbeans.version` must track the NetBeans Platform release the *current downloadable
  VisualVM distribution* actually bundles — not the latest `org.netbeans.api:*` release on Maven
  Central.** `org.netbeans.api` publishes every NetBeans release going back years, so it happily
  resolves whatever version the `pom.xml` asks for even when that's newer than what any real
  VisualVM install ships with. Building against a too-new version doesn't fail the build; it
  fails silently until a user tries to install the `.nbm`, where VisualVM's plugin installer
  rejects it with a wall of "The plugin `<X> API` is requested in version >= A.B but only C.D was
  found" warnings, one per NetBeans module (Settings, UI Utilities, Module System, Base
  Utilities, Lookup, Utilities, Window System, etc.) — this happened once already, from an
  initial (wrong) guess of `RELEASE300`. Confirmed by diffing each flagged module's
  `OpenIDE-Module-Specification-Version` (`unzip -p <artifact>-<RELEASE>.jar META-INF/MANIFEST.MF`)
  across candidate `RELEASE2xx`/`RELEASE3xx` versions from Maven Central against the "only C.D
  was found" numbers in the actual warning text — `RELEASE220` was the exact match for every
  single flagged module. If this drifts again after a VisualVM upgrade, the same diffing approach
  is the fix — don't just bump to "latest."
- Pin `visualvm.version` (`org.graalvm.visualvm.api`) the same way: match the plugin API version
  actually bundled with the target VisualVM release, not necessarily the newest on Central.
- Because of the above, **this module builds against plain JDK 11** (this repo's baseline) —
  `RELEASE220`'s jars are Java 8 bytecode (class-file major version 52), not JDK 21 like
  `RELEASE300`'s were. No special JDK carve-out needed here, unlike `stress-test/`.
- All VisualVM/NetBeans Platform dependencies resolve straight from Maven Central — no
  IDE-registered platform, no bundled/local-only jars, no custom `<repositories>` needed:
  - `org.graalvm.visualvm.api:org-graalvm-visualvm-{application,application-views,core,host,tools}`
    — the modern (GraalVM-era) VisualVM plugin API. Source imports were mechanically rewritten
    from the old `com.sun.tools.visualvm.*` packages to `org.graalvm.visualvm.*` to match — a
    pure package rename, no API/behavior changes were needed.
  - `org.netbeans.api:org-openide-{awt,modules,util,util-lookup,util-ui,windows}` and
    `org-netbeans-modules-settings` — the underlying NetBeans Platform APIs, published by the
    Apache NetBeans project. `org-openide-util-ui` (home of `ImageUtilities`) is easy to miss —
    it's a separate module from `org-openide-util` itself; `nbm-maven-plugin`'s bytecode
    verification (`verifyRuntime`, default `fail`) catches a missing one at build time with a
    "Class usages: [...]" error rather than failing silently at runtime.
  - `org.jfree:jfreechart:1.0.19` / `org.jfree:jcommon:1.0.24` — same `org.jfree.chart.*`/
    `org.jfree.data.*` package API as the jars this project used to bundle locally
    (`jfreechart-1.0.14.jar`/`jcommon-1.0.17.jar`); deliberately pinned to the 1.0.x line, not
    the current `1.5.x` line, which restructured/dropped the `jcommon` split this code depends
    on.
  - `org.perfmon4j:perfmon4j:<version>:jar:monitor-agent` — the RMI client stubs
    (`RemoteManagementWrapper` et al.), replacing the ancient bundled
    `perfmon4j-1.2.0-*-monitor-agent.jar`. Pin this to an actually-released version (check
    `readme.txt`), not the in-repo `-SNAPSHOT`, since this module builds standalone and doesn't
    see the reactor's local build.
- **The manifest wiring** (`nbm-maven-plugin` 14.5): the `nbm:manifest` goal (bound to
  `process-classes`) generates most of the `OpenIDE-Module-*` manifest into
  `target/classes/META-INF/MANIFEST.MF`, merging in anything from `sourceManifestFile`
  (`src/main/nbm/manifest.mf` by convention) without overwriting it — that's where
  `-Install`/`-Layer`/`-Localizing-Bundle`/`-Requires` are hand-supplied here, since the plugin
  doesn't auto-detect those from bytecode/resource scanning. For that generated manifest to
  actually land in the packaged jar, `maven-jar-plugin` needs
  `<archive><manifestFile>${project.build.outputDirectory}/META-INF/MANIFEST.MF</manifestFile></archive>`
  — the older `useDefaultManifestFile` shortcut some tutorials mention was removed from
  `maven-jar-plugin` years ago. Skipping this wiring fails late, at the `nbm:nbm` goal, with
  "Missing manifest tag OpenIDE-Module" rather than at compile time.
- `<codeNameBase>org.perfmon4j.visualvm</codeNameBase>` is set explicitly in `nbm-maven-plugin`'s
  config — without it, the plugin defaults the code-name-base to
  `${project.groupId}.${project.artifactId}` (`org.perfmon4j.visualvm-plugin`), which doesn't
  match the package the source actually lives in.

## Commands & Scripts
- **Build**: `mvn clean install` from `visualvm-plugin/` — produces
  `target/nbm/visualvm-plugin-<version>.nbm`.
- **Manual verification** (can't be scripted in CI): download a real VisualVM 2.2.x distribution,
  install the built `.nbm` via Tools → Plugins → Downloaded → Add Plugins, relaunch, attach it to
  a JVM running with perfmon4j's `ExternalAppender`/RMI listener enabled, and confirm the
  "Perfmon4j" tab appears on that application and can browse live monitors.
